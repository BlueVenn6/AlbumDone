# Mobile Known Risks Before First Release

## 风险 1：AsyncStorage 原生链接未真正生效

- [ ] 风险点：`@react-native-async-storage/async-storage` 原生模块未在真机正确链接
- [ ] 触发场景：Android 本地安装后首次保存设置
- [ ] 表现：持久化无效、报 `NativeModule` 缺失
- [ ] 临时规避办法：重新执行原生依赖同步，Android 重新 clean/build，iOS 重新 `pod install`
- [ ] 后续正式修复方向：补稳定的 CI 原生校验与真机 smoke test，锁定依赖版本

## 风险 2：iOS `ph://` 读取仍需真机验证

- [ ] 风险点：`ph://` 元数据与缩略图读取兼容性仍依赖真机
- [ ] 触发场景：iPhone 从系统照片库分页读取照片
- [ ] 表现：图片不显示、分页异常、流程中断
- [ ] 临时规避办法：优先用常规相册图片测试，记录失败 URI 与机型
- [ ] 后续正式修复方向：补更稳定的 iOS 原生 asset 读取桥接与回归用例

## 风险 3：Android `content://` 在不同 ROM 上兼容性不一致

- [ ] 风险点：不同 ROM 的内容提供器返回格式和权限边界可能不同
- [ ] 触发场景：小米、华为、三星等不同 ROM 读取系统相册图片
- [ ] 表现：同样流程在部分机型成功、部分机型失败
- [ ] 临时规避办法：首测优先覆盖 2 个以上 ROM；失败时记录 URI、相册来源、系统版本
- [ ] 后续正式修复方向：补 content resolver 兼容层与机型回归矩阵

## 风险 4：`hasApiKey` 与“当前会话可调用 AI”之间可能误导

- [ ] 风险点：UI 可显示“已配置”，但运行时内存 key 可能已丢失
- [ ] 触发场景：重启 App 后未重新输入 key，就直接执行 AI 链路
- [ ] 表现：设置页看起来已配置，但 AI 调用失败
- [ ] 临时规避办法：首测明确要求重启后重新输入会话内 key，再验证 AI 链路
- [ ] 后续正式修复方向：接入安全存储，并区分“已保存非敏感配置”和“当前会话 key 可用”

## 风险 5：运行时内存 key 的重启后 UX 仍偏弱

- [ ] 风险点：当前为避免明文持久化，key 只存在运行时内存
- [ ] 触发场景：用户杀掉应用或系统回收进程后再次进入设置页
- [ ] 表现：需要重新输入 key；若提示文案不足，用户可能误判为保存失败
- [ ] 临时规避办法：首测时把“重启后需重新输入 key”写入测试说明
- [ ] 后续正式修复方向：接入 Keychain / Keystore 等安全存储并补明确提示
